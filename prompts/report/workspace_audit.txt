## Instructions

Use your tools systematically to examine the workspace:

1. **Project structure** — Use `analyze_module_structure` and `list_directory`
   to understand the codebase layout, module sizes, and dependency graph.

2. **Documentation vs reality** — Use `check_docs_vs_implementation` to find
   planned features, then use `search_codebase` and `read_file` to verify
   whether each is actually implemented.

3. **Code health** — Use `scan_todos` to find TODO/FIXME/HACK markers.
   Use `check_imports` on key modules to find broken imports.
   Use `search_codebase` for `NotImplementedError`, `pass  #`, `raise$`.

4. **Test health** — Use `run_tests` to get the current test status.
   Note failures, skips, and areas with no test coverage.

5. **Git activity** — Use `read_git_log` and `read_git_branch` to
   understand recent work, active branches, and momentum.

6. **Quality issues** — Search for:
   - Duplicated patterns across files (copy-paste code)
   - Security concerns (hardcoded secrets, open permissions)
   - Missing error handling at boundaries
   - Dead code or unreachable branches

## Report Format

Produce a structured markdown report with these exact sections:

```
# Workspace Audit Report
Generated: [date]

## Summary
[2-3 sentence overview of project health]

## Critical Issues

- Issue: [description]
  File: [path:line]
  Severity: CRITICAL | HIGH
  Fix: [what to do]

## Unfinished Features
[Features that are planned/documented but not implemented or only partial]
- Feature: [name]
  Source: [which doc mentions it]

  What exists: [if partial, what's done]
  What's missing: [specific gaps]

## Code Quality
[Patterns that could be improved, duplication, inconsistencies]
- Issue: [description]
  Where: [files involved]
  Impact: [why it matters]
  Suggestion: [how to fix]

## Test Coverage
[Test results summary, areas without coverage]
- Passing: N tests
- Failing: N tests (list them)
- Missing coverage: [modules with no tests]

## TODOs and Debt
[TODO/FIXME items found in code, grouped by area]

## Architecture Notes
[Module structure, dependency patterns, notable design decisions]

## Recommended Priorities
[Ordered list of what to work on next, with effort estimates]
1. [action] — [reason] — [effort: Xmin/Xhr/Xday]
```

Be specific. Reference file paths and line numbers. Do not pad with
generic advice — every item must be grounded in what you observed.

Return ONLY the markdown report, no conversational text around it.
